## Vision

### 10x Check

diffprompt is already a good local evaluator. The 10x unlock is **usability inside the environments real teams work in**: corporate networks where all LLM traffic goes through an internal proxy with bearer-token auth, `localhost:11434` doesn't resolve, and `OPENAI_API_KEY` is not a thing. Shipping that unblocks one real team today (the operator's), and every other team behind a corp proxy the day after.

The other 10x is **time-to-first-eval**. A new user today: `diffprompt init` scaffolds empty files; they stare at `baseline.md`, `candidate.md`, and a 1-line `cases.jsonl`; they close the tab. Fix: `diffprompt quickstart` is a 10-second loaded demo (mock provider, preloaded cases, instant win/loss grid). `diffprompt init --wizard` uses the configured judge to propose starter baseline + candidate + 10 synthetic cases from a single-sentence description.

### Platonic Ideal

A developer behind any LLM gateway — OpenAI direct, Groq, OpenRouter, Ollama, corp proxy, Vertex-over-HTTP, Bedrock-compatible — configures once in `diffprompt.config.json`, points at their secrets file or env var, runs `diffprompt run`, gets a calibrated pairwise verdict. Zero custom adapters to write. The config surface treats the corporate proxy the same way it treats OpenAI: "provider = OpenAI-chat-completions-compatible; here's my base URL; here's my auth header."

### Felt Experience

You clone the repo Monday morning at Expedia. `diffprompt quickstart` prints a green win/loss grid in 10 seconds on mock. You `diffprompt init --wizard --describe "refund intent classifier"`, answer three questions, get a scaffold with 10 judge-generated cases + a reasonable first prompt. You add one stanza to `diffprompt.config.json` pointing at `corp-proxy` with `gpt-5.1`, `export DIFFPROMPT_CORP_PROXY_KEY=$(cat ~/.expedia/proxy-token)`, run `diffprompt run --mock off`, and get your first real comparison. Total elapsed: under 15 minutes. No PR blocked on a platform team shipping something.

## Security — Corp-Proxy Handling

Non-negotiable rules for the operator-directive adapter:

1. **No URL in committed code.** The Expedia corp-proxy base URL and model id are not defaulted anywhere in shipped source. They come from `diffprompt.config.json` (local, repo-specific) or environment. The *adapter* is generic ("OpenAI-compatible proxy with custom headers"); the *URL* is user-supplied config.
2. **No token in config files.** Auth key is only read from:
   - An environment variable (config field: `"keyEnv": "DIFFPROMPT_CORP_PROXY_KEY"`), **or**
   - A gitignored secrets file at a user-specified path (config field: `"keyFile": "~/.diffprompt/corp-proxy.key"`).
   Never from `"key": "sk-..."` inline. If a future user tries to paste a key inline, the config loader rejects with a loud error.
3. **Custom headers as first-class config.** The corp proxy requires `x-client-app: generative-ai-proxy`. Instead of a hardcoded Expedia header, the provider block accepts `"headers": { "x-client-app": "generative-ai-proxy" }` — portable to any corp proxy with bespoke header requirements (Azure APIM, Kong, internal gateways).
4. **`.gitignore` guard.** `diffprompt init` scaffolds an entry in `.gitignore` for the default secrets path so fresh projects don't accidentally commit tokens.
5. **Redact in logs.** `diffprompt run --verbose` strips any header value whose key matches `/auth|token|key|secret/i` before printing.

Documentation: `docs/guide.md` gets a "Providers → Corporate / self-hosted proxies" section with a complete worked example using Expedia's proxy as the template (with the token pulled from `$DIFFPROMPT_CORP_PROXY_KEY`, never inlined).

## Risks

1. **Operator assumes corp-proxy provider works identically to OpenAI direct.** GPT-5.1 via the Expedia proxy may reject long system prompts, cap output tokens differently, or hang on concurrency > 1. Mitigation: wire a quick `diffprompt providers test corp-proxy --hello` smoke command (one-sentence prompt, expects a response) so the first failure is diagnosable, not a mystery in the run grid. ~15 min.
2. **Wizard judge-generated cases look plausible but are wrong.** The user gets 10 cases with expected answers they didn't write — may ship a prompt that passes the eval but fails production. Mitigation: wizard output is marked `"_autogenerated": true` in the JSONL and the PR comment / report both flag unaudited autogen cases in the banner ("N of M cases are autogenerated — review before trusting the verdict"). ~10 min incremental.
3. **Named-provider config breaks existing users.** Current `ModelId` prefixes (`openai/`, `groq/`, `openrouter/`, `ollama/`) are hardcoded in docs, tests, user configs. Mitigation: the four existing names stay as *built-in registered providers* on load — user `providers: [...]` adds to the list rather than replacing it. Hardcoded prefixes continue to work without any config change.
4. **CSV seed adapter hits a file with unsanitized user content containing commas, quotes, newlines.** Mitigation: use a battle-tested CSV parser (not regex), run the existing PII heuristic on imported rows the same way Langfuse/Helicone/LangSmith do.
5. **Phase 2 hibernates forever.** Usage-first is the right call today; risk is the hosted surface never ships and diffprompt stays a niche CLI. Mitigation: 4-week check-in. If v1.1 has <5 external users and no corp-proxy success story by 2026-05-22, reopen Phase 2 conversation regardless of adoption metric.
