## Why this plan supersedes v1.1

v1.1 shipped cleanly. Named providers unblock the operator's corp-proxy; wizard + quickstart cut time-to-first-eval. The operator now has the tool running against real traffic and the next question is obvious: **"Is this good enough to replace LangSmith for prompt + finetune workflows?"** Today the honest answer is no on three axes:

1. **Eval depth.** rubric is pairwise-only. A real eval system needs composable metrics — exact match, regex, JSON-schema validity, length bands, judge verdict — running in one sweep with one row per cell.
2. **Async execution.** `rubric run` is blocking and non-resumable. A 500-cell sweep at 2s/cell is 16 minutes of Ctrl-C-unsafe waiting. LangSmith's hosted experiments just work.
3. **Finetune loop.** The interesting loop — "generate pairs → winners become SFT training data → launch 4 fine-tunes → auto-eval winners" — doesn't exist in LangSmith either. This is our wedge, not our gap.

v1.2 closes 1 and 2, opens 3 as a first-class surface.

## Vision

### 10x check

rubric becomes the **local-first eval + finetune loop for prompt and model engineering**. Every artifact is a file on disk — dataset, prompt, config, run manifest, finetune spec, evaluator. Every result is a reproducible, diffable, git-tracked record. LangSmith is cloud-hosted, framework-coupled, UI-centric; rubric is text-first, local-first, git-tracked, and composable. The 10x is that you can run a prompt sweep, take the winning cases, fine-tune four candidate models on them, and auto-evaluate which one beats baseline — all from three `rubric` commands and three YAML files, with no hosted service, no vendor lock-in, and no UI that breaks when the browser tab crashes at 3am.

### Platonic ideal

```bash
# 1. Eval your prompt change across a solid set of metrics
rubric run --detach                       # returns run id; CI-safe, Ctrl-C safe, resumable
rubric runs wait r-01JXY...                # block until done
rubric runs show r-01JXY...                # pairwise winrate + exact-match + schema-valid + latency

# 2. Turn winners into training data
rubric finetune prepare --from-run r-01JXY... --out finetunes/sft.jsonl

# 3. Launch multiple fine-tunes in parallel
rubric finetune launch --all               # 4 jobs, OpenAI SFT, parallel
rubric finetune status                     # grid view

# 4. Auto-eval the trained models against baseline
rubric finetune eval                       # registers ft:gpt-4o-mini:YOURORG:sft-v1 etc
                                          # runs the standard pairwise eval, one report per model
```

### Felt experience

You're debugging a classifier at 9pm. You edit `prompts/candidate.md`. `rubric run --detach` returns instantly; you switch to a different terminal and keep working. 90 seconds later `runs wait` clears and you see the matrix: candidate beats baseline 34–8–8, exact-match on the gold-label field is 92% vs 78%, schema-valid 100% vs 94%, mean latency +30ms. You `rubric runs diff r-current r-yesterday` and see the regression was a latency hit, not quality. Ship.

Three weeks later you have 2,000 winning cases from production prompt runs. `rubric finetune prepare` converts them to SFT format in five seconds. `finetunes.yaml` declares four jobs: two base models × two hyperparameter sets. `rubric finetune launch --all` fires all four off OpenAI's API. An hour later `finetune status` shows all green. `rubric finetune eval` produces four pairwise reports — two of the fine-tunes beat baseline, two don't. You pick the winner, pin its model id in `rubric.config.json`, ship. Total UX: seven commands across three files. No dashboards. No signup. Everything git-tracked.

## Architecture

### Eval surface (additive)

Today: one `Judge` produces one `JudgeResult` per cell. Tomorrow:

```ts
// New: shared/src/evaluators.ts
export interface Evaluator {
  readonly name: string;
  readonly kind: 'judge' | 'metric';
  evaluate(ctx: EvalContext): Promise<EvaluatorResult[]>;
}
export interface EvaluatorResult {
  metric: string;              // e.g. "pairwise.winner", "exact_match", "json.valid"
  value: number | string;
  pass?: boolean;
  reason?: string;
}
```

The existing `Judge` adapts into an `Evaluator` with `kind: 'judge'` emitting `pairwise.winner`, `pairwise.reason`. New evaluators are `kind: 'metric'`. `CellResult` gains `evaluations: EvaluatorResult[]` (optional; backward-compat when empty).

**Config:**
```json
{
  "judge": { "model": "openai/gpt-4o", "criteria": "default" },
  "evaluators": [
    { "type": "exact-match", "field": "expected" },
    { "type": "json-schema", "schema": "./schemas/refund.json" },
    { "type": "length", "min": 20, "max": 200 }
  ]
}
```

Empty `evaluators` = today's behavior, exactly. No migration needed.

### Run registry

```
~/.rubric/runs/<runId>/
  manifest.json    { id, startedAt, finishedAt?, configHash, datasetHash,
                     promptsHash: { baseline, candidate }, models[], status }
  cells.jsonl      append-only; one line per completed cell
  log              stderr/progress capture
  lock             present while a worker owns the run
```

`runId` = ULID. `status` ∈ `{pending, running, complete, failed, abandoned}`.

`rubric runs list` — tabulates recent runs with status, models, winrate.
`rubric runs show <id>` — renders the same summary as `rubric run` post-hoc.
`rubric runs diff <a> <b>` — prints summary delta + links to both HTML reports.
`rubric runs rerun <id>` — re-executes with identical config (re-reads prompts/dataset from current disk, not the manifest — so `rerun` is "same config, current prompts").
`rubric runs resume <id>` — re-executes only cells missing from `cells.jsonl`.
`rubric runs wait <id>` — polls until status ≠ running.
`rubric runs status <id>` — prints status + progress count.

### `--detach`

`rubric run --detach`:
1. Writes manifest with `status: pending`.
2. Forks a detached child process running the same `runRun(...)` with `--resume <id>` semantics, lock acquired.
3. Parent prints the run id and exits 0.
4. Child streams cells to `cells.jsonl`, writes final `status: complete|failed` on exit.

Uses `child_process.spawn` with `detached: true, stdio: ['ignore', logFd, logFd]`. No hosted daemon.

### Finetune orchestration

```
finetunes.yaml              # user-editable, git-tracked
  jobs:
    - name: sft-gpt4o-mini-v1
      provider: openai
      baseModel: gpt-4o-mini-2024-07-18
      trainingData: finetunes/sft.jsonl
      validationData: finetunes/val.jsonl
      hyperparameters: { n_epochs: 3 }
    - name: sft-gpt4o-mini-v2
      provider: openai
      baseModel: gpt-4o-mini-2024-07-18
      trainingData: finetunes/sft.jsonl
      hyperparameters: { n_epochs: 5 }
```

State lives in `~/.rubric/finetunes/<name>/state.json`:
```json
{ "name": "sft-gpt4o-mini-v1",
  "providerJobId": "ftjob-abc123",
  "status": "succeeded",
  "fineTunedModel": "ft:gpt-4o-mini-2024-07-18:yourorg:sft-v1:abc",
  "trainingFileId": "file-xyz789",
  "createdAt": "...", "finishedAt": "..." }
```

Commands:
- `rubric finetune init` — scaffold `finetunes.yaml` with one example job.
- `rubric finetune prepare --from-run <runId> [--winners-only] --out finetunes/sft.jsonl` — converts pairwise-winner cells to OpenAI SFT `messages` format.
- `rubric finetune launch [--all | --job <name>]` — uploads training file, creates fine-tune job, persists state. Concurrent across jobs.
- `rubric finetune status` — polls and displays grid.
- `rubric finetune eval [--against prompts/baseline.md]` — for every succeeded job, runs `rubric run` with the fine-tuned model id as a `candidate` model, producing one pairwise report per job.

**Provider plumbing:** OpenAI fine-tuning uses the standard key (`OPENAI_API_KEY`) and the existing `createOpenAIProvider`. The fine-tune API is separate from the chat completions surface, so we add `packages/shared/src/finetune/openai.ts` with a thin wrapper (`createFineTune`, `getFineTune`, `uploadTrainingFile`).

## Security

- Run registry lives at `~/.rubric/` — same scope as existing secrets files. Never checked into repo.
- Fine-tune state never contains API keys — only provider job ids and returned model ids.
- `--detach` child inherits only explicitly whitelisted env vars: `OPENAI_API_KEY`, `GROQ_API_KEY`, `OPENROUTER_API_KEY`, and user-declared `keyEnv` names. Nothing else.
- Training data is read from local paths; never uploaded to anywhere except the provider the user explicitly declared in `finetunes.yaml`.
- `rubric finetune launch` prints the provider + base model + training-file-size summary before uploading; `--yes` required for non-interactive CI use. No silent uploads.

## Error & Rescue Map

| Failure | Detection | Rescue |
|---------|-----------|--------|
| Detached child crashes mid-run | `status` stays `running`, lock PID dead | `rubric runs resume <id>` — skips cells already in `cells.jsonl` |
| Fine-tune job fails at provider | Provider returns `status: failed` on poll | Surface `error` field from provider; user inspects, fixes data, `rubric finetune launch --job <name>` re-runs that one |
| Eval mid-run but prompts changed on disk | Prompt hash mismatch between manifest and current disk | `rubric runs show` prints "prompts modified since run" banner; `runs rerun` refuses unless `--force` |
| Two workers race on same run id | Lock file contention | Second worker exits 1 with "run <id> is already running (pid N)" |
| Evaluator throws | Caught per-cell; `evaluations[]` entry carries `error` field | Run continues; summary shows N cells with evaluator errors |
| `finetunes.yaml` declares a model that doesn't fine-tune | Provider returns 400 | Surface verbatim; user fixes base model id |
| User runs `finetune eval` before jobs complete | State file says `status: running` | Skip those jobs, print a banner listing pending ones |

## Performance

- Run registry writes are append-only JSONL — O(1) per cell. No locking between writes (only one worker owns the run).
- Manifest + cells parsed lazily on `runs list` (stat + first-line parse, not full scan).
- `--detach` adds ~20ms for spawn; `wait` polls at 500ms intervals.
- Fine-tune polling: 10s interval, exponential backoff on transient errors, honored by the OpenAI SDK.

## Observability

- Every run id ends up in the manifest with timestamps — after-the-fact audits are one `cat ~/.rubric/runs/*/manifest.json | jq` away.
- Log stream at `~/.rubric/runs/<id>/log` captures every stderr write from the worker — operator-level debuggability without any metrics pipeline.
- New: `rubric runs stats --last 7d` (v1.3) — aggregates cells/day, mean latency, cost trendline from the registry. Noted, out of scope here.

## Long-term trajectory

This increment moves rubric from "pairwise diff tool" to "eval loop + finetune loop." The two loops share the run registry, which is the foundation for Phase 2's hosted surface when it unfreezes — a hosted rubric is "run registry backed by Postgres + S3 instead of `~/.rubric/`." The CLI and hosted surfaces speak the same manifest format.

The story becomes: **"We are LangSmith, except every artifact is a file you can commit, and we do fine-tunes LangSmith doesn't."** That is a differentiated pitch against a hosted incumbent, and it lands with the exact cohort that rejects hosted tools on principle (corp IT, compliance-heavy teams, power users who live in git).
