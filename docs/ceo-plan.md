## Vision

### 10x Check

diffprompt is not just a local CLI ŌĆö it's a **two-layer product**: a local tool for builders who want full control (`npx diffprompt serve`), plus a hosted preview layer (`diffprompt.dev`) that turns every comparison into a shareable URL. You paste two prompts, see the live diff across 5 models using demo keys, tweet the link, your followers "Fork this diff" into their own local install. The tool *is* the marketing. Every diff a user shares is a distribution event.

### Platonic Ideal

diffprompt becomes **infrastructure every serious AI repo checks in**. A `prompts/` directory, a `diffprompt.config.ts`, a GitHub Action. Open a PR that touches a prompt ŌåÆ the bot comments: "v2 wins 42/50, ties 5, loses 3, judge agreement 87%, full diff at diffprompt.dev/pr/1234". Reviewers ship-or-reject in the PR, not a separate tab. The README badge shows live baseline win-rate. The drift detector watches for model updates silently breaking your prompts. This is "git for prompts" ŌĆö not a dev tool, a primitive.

### Felt Experience

You're an AI dev iterating on a support-bot prompt. You edit v2, hit Run, watch the grid fill in 30 seconds across 3 models ├Ś 50 cases. 41 wins, 6 losses. Click a losing row, see v1 said "contact support" where v2 hallucinated a phone number ŌĆö judge agrees. Hit "Steelman this case," Claude proposes a targeted fix. Run again. 47 wins. Open a PR, the bot comments the summary, your teammate approves. Ship. Your README badge ticks up from 82% ŌåÆ 94%. Four weeks later you get a drift alert: "Claude 4.8 dropped, your baseline changed on 3 cases." You investigate. You stay one step ahead of the regression before a customer notices.

## Judge Calibration (launch-gate item ŌĆö pulled in from v2)

The public surfaces added by SCOPE EXPANSION (PR bot comment, README badge, drift detector) make the judge load-bearing on day one. Shipping an uncalibrated judge behind a PR comment invites the first-screenshot-kills-the-narrative scenario. Calibration moves into v1:

- **`diffprompt calibrate` CLI.** User labels 50+ pairs; tool computes judge-vs-human agreement rate. Stored per-repo in `prompts/_calibration.json` (committed to git alongside prompts).
- **Calibration-aware public surfaces.** PR bot comment and README badge each read `_calibration.json` and render differently based on state:
  - No calibration file ŌåÆ bot comment prefixed "ŌÜĀ’ĖÅ unverified judge ŌĆö treat as signal, not verdict" and badge shows a grey "unverified" variant.
  - Calibration file present ŌåÆ bot comment shows agreement-rate footer ("judge agrees with human labels on 87% of pairs") and badge flips to colored confidence band.
- **In-UI calibration flow.** `diffprompt serve` surfaces a "Calibrate judge" tab with side-by-side labeling UI ŌĆö 5 minutes of clicking, not a data-labeling project.
- **Cost.** ~1 CC day added. Slips Week-5 polish (steelman button / why-failed drawer) without drama ŌĆö they were already cuttable.

## Stack Decision: Workers + Durable Objects

Cloudflare Workers free tier has a 30s CPU wall, paid tier has a 5min wall. A 50-case ├Ś 3-model sweep even at concurrency=4 can blow past both. Resolution:

- **Workers** handle HTTP, auth, webhooks, badge rendering ŌĆö fast, stateless, fits the CPU budget.
- **Durable Objects** run the eval engine. A DO per run spawns per-cell LLM subrequests and streams results back via WebSocket to the browser. DOs don't hit the Worker CPU wall for I/O-bound work (LLM calls are I/O, not CPU).
- **Week-1 spike (~half day, upgraded from 2 hrs)** validates the DO eval-runner pattern with the actual workload before committing. Fallback: Fly.io VPS for the eval layer, Workers for everything else.
- **R2** stores run artifacts (already in the plan). **D1** stores run metadata + judge verdicts + calibration state.

## Dataset Bootstrap: Langfuse Import (launch-gate item ŌĆö pulled from v2)

Hand-labeling 50 pairs to calibrate the judge is the calibration flow's weakest point. Langfuse (OSS LLM observability, widely adopted) exports JSONL where each row contains `input`, `output`, and a `feedback` field with `{score: number, comment?: string}` or thumbs-up/down. That feedback field IS labels. Users who already run Langfuse get a free calibration set.

- **`diffprompt seed --from-langfuse <export.jsonl>`.** Parses Langfuse export format. Maps:
  - Rows with positive feedback ŌåÆ candidate "good" examples for the test set.
  - Rows with negative feedback ŌåÆ "regression probes" (cases where the baseline is known to have failed; v2 should improve).
  - Feedback score Ōēź threshold ŌåÆ human label for the calibration flow, feeding `diffprompt calibrate` without manual clicking.
- **Dedup + sampling.** 10k-row exports are common; sample down to a user-specified N (default 50) with stratified sampling across feedback bands.
- **PII scrubbing on import.** Reuse the same regex nudges from the sharing flow. Warn, don't block.
- **Pitch line.** "Paste your Langfuse export ŌåÆ calibrated judge in 5 minutes."
- **Cost.** ~half CC day (parser + mapping + stratified sampler). Format adapter pattern leaves room for Helicone / LangSmith / OpenAI-logs importers in v1.1.

## Comparison Modes: Prompts vs Models (launch-gate item)

The core eval engine already runs a `cases ├Ś prompts ├Ś models` grid. v1 exposes two UI modes on top of that grid:

- **Compare prompts** (default): baseline prompt vs candidate prompt across N models. The original v1-vs-v2 loop.
- **Compare models**: lock one prompt, vary models. Test "should we upgrade from Claude 4.6 ŌåÆ 4.7?" or "GPT-5.4 vs Gemini 2.5 on this prompt?" ŌĆö a decision every AI team makes monthly.
- **Shared plumbing = drift detector.** Drift detection = scheduled "compare models" run on a locked prompt with previous-version results as baseline. Shipping comparison mode in v1 means v1.1's drift detector is nearly free (just add the cron).
- **UI.** Single toggle in the run setup: "Vary [prompts | models]." No new nav, no new docs page. The judge rubric tweaks slightly ("pick the model whose output best accomplishes the goal" vs "pick the prompt...").
- **Cost.** ~half CC day (UI toggle + rubric variant + result framing).

## Abuse & Cost Containment (launch-gate item)

Running a hosted sandbox with free LLM demo keys is the single highest-risk decision in this plan. One bad actor can exhaust the budget in hours. In-scope at launch:

- **Provider-TOS review.** OpenAI, Anthropic, Google all restrict redistributing API access. Before launch, confirm each provider's terms allow our "anonymous free tier" ŌĆö if not, switch to "bring your own key in browser localStorage" for the anonymous tier. Blocker for `diffprompt.dev` launch, not for local-only launch.
- **Daily spend cap.** Global $X/day budget (user-configurable per env var), kill-switch when exceeded, waitlist page displayed to new sandbox users until next day.
- **Per-IP rate limits.** 10 runs/day/IP on the anonymous sandbox. Redis-backed counter.
- **Content caps.** Max prompt length 4k chars, max dataset size 20 cases on the sandbox tier. Full-size runs require signed-in + own-keys.
- **Content moderation.** Run pasted prompts through a cheap safety classifier before sending to demo-key provider. Reject obvious abuse (prompt injection probes, illegal content).
- **PII/secret scrubbing.** Warn (don't block) when paste contains patterns matching API keys / emails / phone numbers. Nudge users to redact before sharing.
- **Legal pages.** Privacy policy, TOS, DMCA contact. Ship behind `diffprompt.dev/legal/*` at launch.
- **Kill switch.** Single env var flip disables the anonymous sandbox entirely without taking down signed-in users or the GitHub App.
- **Budget alerting.** Slack webhook to builder's personal Slack on 50%, 80%, 100% of daily cap.

## Open Architecture Decisions Resolved

- **PR bot: comment-only, not merge-gate.** The bot posts the win/tie/loss summary. Repo admins can configure branch protection to require a passing check (separate, optional ŌĆö not our code). This matches how tools like Chromatic and Vercel Preview work. The deferred "`--fail-on-regress` CLI gate" is **un-deferred** and ships as a CLI flag in `diffprompt run` (the bot uses it internally) so users who want hard gates get one.
- **Badge cache invalidation.** Bot updates a row in the hosted DB on every PR run. Badge endpoint serves a 60-second-cached SVG keyed on `<org>/<repo>@<branch>`. Stale-while-revalidate via Cloudflare cache.
- **Drift detector budget.** Free tier: 1 re-run per month per installed org, on a fixed small dataset (Ōēż20 cases). Anything larger requires own-keys. Surfaces as "add your own API keys to run drift on your full dataset" upsell path.
- **Fork UX ŌĆö two paths.** Shared URLs expose two buttons:
  - **Remix in browser (primary).** One click clones the run into an editable sandbox that reuses demo keys and rate limits. Zero setup. This is the viral path.
  - **Fork to local (secondary).** `npx diffprompt pull <id>` for serious users willing to install Node + add keys. Labeled as the "serious track."
  - Anonymous runs: both paths allowed.
  - Signed-in persisted runs: public by default (owner can mark private). Private URLs require the owner to share an invite token.
  - If pulled/remixed comparison used demo keys, downstream runs against user's own keys may produce different results ŌĆö documented clearly.
- **Baseline storage.** `prompts/_baseline.md` is the locked version. `diffprompt lock` copies candidate ŌåÆ baseline. Baseline lives in git alongside the prompts. Hosted layer reads from the GitHub repo via App permissions; no cloud-side baseline storage for GitHub-linked repos.
