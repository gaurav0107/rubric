## Problem Statement

Builders iterate on prompts constantly. Every change risks silent regressions: v2 is better on the cases you looked at, worse on the cases you didn't. Today you either (a) eyeball a handful of outputs and ship on vibes, or (b) glue together an eval script each time. There is no lightweight tool that makes "is v2 actually better than v1" the primary interaction — you have to build the dataset, build the runner, build the comparison view, every time. rubric is a local-first OSS tool that makes pairwise prompt comparison across multiple models the default motion: start with a baseline, propose a new prompt, see a live side-by-side diff with a calibrated judge verdict, ship or reject based on evidence.

## What Makes This Cool

Pairwise diff is the core verb, not a feature. Existing tools (Promptfoo, OpenAI Evals, Langfuse, Inspect AI) show pass/fail tables — useful but flat. rubric makes the v1-vs-v2 delta the center of the screen: every test input renders as a side-by-side pane with v1 on the left, v2 on the right, judge verdict in the middle, aggregate win-rate at the top. Edit v2, hit Run, and watch the grid re-stream: first responses arrive in ~2s, full 50-case × 3-model sweep completes in ~30-60s depending on provider latency. Not keystroke-live — honest expectation is "save, run, watch progress fill in under a minute." That alone is faster than any existing tool makes prompt comparison feel.

Paired with that: a calibrated-trust layer on LLM-as-judge. Label 50+ pairs yourself, rubric reports judge-vs-human agreement rate and surfaces a score band based on it. Most tools treat judge scores as ground truth. Ours treats them as signal with measured noise. **Updated via CEO plan:** calibration pulled into v1 (was v2) because the PR bot and README badge make judge output public from day one — an uncalibrated judge behind a public surface is a narrative-killer on first bad screenshot. See `ceo-plans/2026-04-21-rubric.md` §"Judge Calibration."

## Constraints

- **Local-first.** Runs on the user's machine, writes to `./prompts/*.md` on their disk. No cloud, no signup, no API for the project itself.
- **OSS distribution.** MIT license, ship via npm (`npx rubric serve`) and/or a single-file binary via Bun compile.
- **Multi-model out of the box.** OpenAI, Anthropic, Google, plus any OpenAI-compatible endpoint (Groq, OpenRouter, local Ollama). User brings their own keys.
- **Zero-infra for the user.** The server is the user's machine. No hosted multi-tenant component anywhere.
- **Respect the user's existing project layout.** Drop a `rubric.config.ts` and a `prompts/` folder; don't force a framework.

## Premises

1. OSS builders adopt tools they can `npm i` / `brew install` and hack on locally — hosted web apps with signups get ignored. ✓ Agreed.
2. Pairwise diff UX IS a moat, not just surface — Promptfoo has had years to ship a diff view and hasn't made it the primary interaction. The conviction is that making diff the verb shapes every other design decision (dataset format, judge output schema, UI hierarchy). ✓ Held against challenge.
3. Dataset is the hard part. Without eval inputs, comparison is meaningless. rubric must solve dataset bootstrapping in v1 — either via production log import, LLM-generated synthetic cases, or a minimum-viable "paste 10 examples" UX. ✓ Agreed.
4. Distribution = OSS channels. npm + GitHub Releases + a `README.md` with a demo gif. Not a hosted app you have to run for people. ✓ Agreed.

## Approaches Considered

### Approach A: `rubric` CLI + HTML report

Single binary. `rubric run v1.md v2.md --dataset cases.jsonl --models gpt-5.4,claude-4.7` → generates `report.html`.
- Effort: S (human: 1 week / CC+gstack: ~3 hours)
- Risk: Low
- Pros: Fastest ship, zero infra, perfect for an HN post.
- Cons: Dataset problem unsolved (user brings their own JSONL); judge uncalibrated; static report loses the live-diff whoa.

### Approach B: Library + opinionated CI-gate workflow

Core library + CLI + bisect-for-prompts workflow. CI hook exits non-zero on regressions. Dataset bootstrapping via log import. Calibrated judge.
- Effort: L (human: 3-4 weeks / CC+gstack: ~1.5 days)
- Risk: Med
- Pros: Real workflow moat; the "git hook for prompts" narrative; calibration differentiates.
- Cons: Bigger surface; first-release polish pressure; dataset UX can eat the whole first month.

### Approach C: Local server + live diff playground (CHOSEN)

`npx rubric serve` opens localhost:3000. Edit v1 and v2 in-browser, results stream as side-by-side diffs across your dataset × models. Prompts saved to `./prompts/*.md` so they're git-tracked. No signup, no cloud, but the UX is hosted-app quality.
- Effort: M (human: 2 weeks / CC+gstack: ~8 hours)
- Risk: Med
- Pros: Keeps the "hosted web app" whoa without the infra tax. Demo gif writes itself. VS Code users get live editing without a separate extension. The CLI is `rubric run` — the server is `rubric serve` — both ship from the same binary.
- Cons: Browser UX bar is higher than a static report. If the server feels clunky, the pitch dies.

## Recommended Approach

**Approach C: Local server + live diff playground.**

**v1 (ships first — ~2 weeks):**
1. `rubric init` — scaffolds `./prompts/v1.md`, `./prompts/v2.md`, `./rubric.config.ts`, `./cases.jsonl` (with 3 example cases).
2. `rubric serve` — localhost UI, edit prompts in-browser, click Run, streaming diff grid.
3. `rubric run` — headless equivalent: reads config, prints summary table, writes `report.html`. CI-friendly subset.

**v2 (later — ~4-6 more weeks):**
4. Dataset bootstrap: `rubric seed --from-logs logs.ndjson` and `rubric seed --synthetic "..." --n 20`.
5. Calibrated judge: `rubric calibrate` — user labels 50+ pairs, tool reports judge-vs-human agreement rate and surfaces it as a score band.
6. CI-gate workflow: `rubric run --baseline main --fail-on-regress` exits non-zero when win-rate drops below threshold.

**Shipping order rule:** UX before the CI-gate workflow. The OSS community stars tools that feel delightful on first use. The CI story (Approach B's moat) comes in v2 once the playground has traction.

### v1 Interaction Model

- **Edit-to-rerun:** Manual. User edits prompt in the browser textarea, clicks "Run" (or ⌘-Enter). No debounce, no auto-fire on keystroke — too costly and too noisy. A cost estimate renders above the Run button: `~300 calls, ~$0.40 est`.
- **Concurrency:** 4 concurrent requests per provider by default, configurable via `RUBRIC_CONCURRENCY`. Respects provider rate limits on 429.
- **Retries:** Per-call exponential backoff, 3 attempts max (1s, 4s, 16s). On final failure, the cell renders as an error with a retry button.
- **Partial results:** Grid cells stream independently. If 47 of 50 cases complete and 3 fail, the win-rate shows `44/47 scored (3 errors)` — never hides the denominator.
- **Cancel:** A "Stop" button aborts in-flight requests without losing already-completed cells.

### v1 Config Schema

`rubric.config.ts`:

```ts
import { defineConfig } from 'rubric';

export default defineConfig({
  prompts: { baseline: './prompts/v1.md', candidate: './prompts/v2.md' },
  dataset: './cases.jsonl',
  models: ['openai/gpt-5.4', 'anthropic/claude-4.7', 'google/gemini-2.5'],
  judge: { model: 'anthropic/claude-4.7-haiku', rubric: 'default' },
  concurrency: 4,
});
```

### v1 Judge

- **Output schema:** `{ winner: "a" | "b" | "tie", reason: string }`. No confidence field in v1 — calibration ships in v2.
- **Default rubric:** "Given the same user input and two candidate responses A and B, pick the response that best accomplishes the user's goal. Consider correctness, completeness, and clarity. Output JSON."
- **Win-rate formula:** `wins / (wins + losses)` — ties excluded from the denominator. Surfaced as `32/47 decided wins, 3 ties, 12 losses` so the user can audit.

### API Keys

Read from `.env` in the project root. Expected vars: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, plus `RUBRIC_BASE_URL_*` for OpenAI-compatible endpoints. No keychain, no cloud sync, no prompting inside the UI — the user owns their keys.

## Open Questions

- **Tech stack — needs a 2-hour spike before committing.** Leaning Bun + SvelteKit + Vercel AI SDK, but `bun build --compile` on a SvelteKit app with `adapter-node` has known gotchas (asset paths, server bundling). If the spike fails, fall back to a thin Bun HTTP entry + static frontend served from embedded assets. Decision must be made before v1 starts.
- **Dataset format lock-in.** JSONL of `{input, expected?, metadata?}` matching the OpenAI Evals rough shape — reduces friction for users migrating from Promptfoo. Confirm before v1 lands.
- **How to handle non-text outputs** (tool calls, structured JSON). v1 scope: text only. Defer tool calls to v2.
- **Provider parity testing.** v1 commits to OpenAI, Anthropic, Google as first-class. Groq / OpenRouter / Ollama work via OpenAI-compatible base URL but are not tested across all response shapes in v1 — documented as "community reports welcome." Parity testing becomes a v2 task.
- **Launch demo dataset — owner needed.** Commit to "customer support ticket classification, 50 cases, labeled for correct category" as the launch gif. Source: generate synthetically via Claude, hand-audit 10 for quality, publish the dataset in-repo so anyone can reproduce the gif.

## Success Criteria

- **v1 (MVP — ~2 weeks):** `npx rubric serve` works on macOS and Linux from a fresh clone. User edits two prompts, clicks Run, sees streaming diff across 50 cases × 3 models in ~30-60s. `rubric run` prints a win-rate table. Ships to npm.
- **Launch signal:** 500 GitHub stars in week 1 after HN post. 10 "I just replaced my eval script with this" mentions on X / Mastodon / BlueSky.
- **Retention signal:** 5 users who return after a week and run it on a new prompt change. Hard to measure without telemetry — ship an opt-in `did this catch a regression? [y/N]` prompt at the end of `rubric run`.
- **v2 (~6 weeks after v1):** CI-gate workflow, calibrated judge, dataset bootstrapping from logs, single-file binary + Homebrew. The "bisect for prompts" narrative.

## Distribution Plan

**v1 (npm-only):**
- **npm package `rubric`** — the single channel. `npx rubric serve` is the first-run command in the README.
- **CI/CD:** GitHub Actions — on tag push, run `bun publish` to npm. That's it.
- **README with demo gif** — the gif is the product. Record using the committed launch dataset.

**v2 (binaries + Homebrew):**
- Single-file binary via `bun build --compile` for mac-arm64, mac-x64, linux-x64. Attach to GitHub Release.
- Homebrew tap. `brew install rubric`.
- Deferred because multi-arch CI + Homebrew adds ~3 days of infra work that doesn't move the needle for an npm-first OSS launch.
