rubric

Pairwise prompt evaluation
for pull requests.

Stop shipping prompt edits blind. v2.2.1 · May 2026
The problem

Prompt changes ship with almost no safety net.

“Looks better” usually means “I tried three examples”. Then a week later a user hits the fourth example and it’s worse than what we had.

Before

Hand-test three cases, ship

  • Edit candidate.md. Spot-check on “a good one”.
  • Merge. Hope.
  • Regressions discovered via a Slack screenshot weeks later.
  • No way to say why the new prompt is better — just vibes.
With rubric

Pairwise sweep, gated on CI

  • Every PR runs baseline vs candidate across a real dataset.
  • An LLM judge picks a winner per case; CI fails on regression.
  • PR comment shows the top regressions inline, with reasons.
  • Override the judge when you disagree — log feeds calibration.
What rubric does

A pairwise sweep across your dataset, judged cell by cell.

Exit code 0 on pass. 1 on judge errors. 2 when the candidate loses more cells than it wins — that’s your CI gate.

Install and try it

Three commands. No account.

Single-file binary for Linux / macOS / Windows. No Node, no Python, no login wall. Grab it from the GitHub release.

01
Download the binary for your platform.
# macOS / Linux (auto-detects arch):
curl -L https://github.com/rubric/rubric/releases/latest/download/rubric-$(uname -s | tr '[:upper:]' '[:lower:]')-$(uname -m | sed 's/x86_64/x64/;s/aarch64/arm64/') -o rubric && chmod +x rubric
02
Scaffold a workspace.
./rubric init
wrote rubric.config.json
wrote prompts/baseline.md, prompts/candidate.md
wrote data/cases.jsonl
03
Run it — live or mocked.
./rubric serve # three-pane local UI on :5174
./rubric run --fail-on-regress # CI-shaped sweep
Config

One JSON file. Everything else lives on disk.

{ "prompts": { "baseline": "prompts/baseline.md", "candidate": "prompts/candidate.md" }, "dataset": "data/cases.jsonl", "models": ["openai/gpt-5.2"], "judge": { "model": "openai/gpt-5.2", "criteria": "default" }, "mode": "compare-prompts", "concurrency": 4 }
prompts Baseline is the control. Candidate is what you’re proposing in the PR.
dataset One JSON object per line. Each needs input; optional expected + metadata pass through.
models provider/model format. Supports OpenAI, Groq, OpenRouter, Ollama, and user-declared proxies.
judge Another LLM that picks per-cell winners. Override criteria with prose or a rubric file.
mode compare-prompts (same model, two prompts) or compare-models (one shared prompt, two models).

The judge isn’t the final word

You disagree. Say so.

The judge is just another LLM. When you think it got a cell wrong, one click records it — and that log becomes the training corpus for the v2.3 calibration model.

Drive it from the CLI or inline in rubric serve. Every override appends to ~/.rubric/overrides/<project>.jsonl. The same file. Same log. CLI and UI round-trip.

From the CLI

rubric disagree case-3/openai/gpt-5.2 \
  --verdict A --reason "judge missed the factual error in B"
✎ case-3/openai/gpt-5.2 → A — “judge missed…”

Or inline in serve

Your verdict Baseline Candidate Tie judge nailed it
You → Candidate · "judge nailed it"

On GitHub

Drop one file into .github/workflows/

The Action downloads the release binary — no npm, no Node setup, no auth beyond your OPENAI_API_KEY secret. ~15 s install, ~40 s for a 50-case sweep.

The comment is idempotent — subsequent runs update the same comment via a hidden marker instead of stacking. Paths trigger on prompt edits only so CI doesn’t run on unrelated changes.

# .github/workflows/rubric.yml on: pull_request: paths: ['prompts/**', 'data/**', 'rubric.config.json'] jobs: eval: runs-on: ubuntu-latest permissions: { pull-requests: write, contents: read } steps: - uses: actions/checkout@v4 - uses: rubric/rubric@v2 with: fail-on-regress: true env: OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
Where this is going

v2.3 learns to score the judge.

v2.2
Radical cut — the pairwise wedge
Stripped finetune, calibrate, history, share, pull, failure clustering. Focused the whole tool on one loop: baseline vs. candidate, gated on CI.
v2.2.1
In-UI overrides, compare-models, top-regressions in PR
Override buttons in serve. Δ column shows which cases moved since last run. PR comment surfaces the three worst losses inline with inputs + reasons.
v2.3
Residual-classifier calibration
Train a small classifier on the override log. Output: per-cell “judge likely wrong” score — surfaced in the comment as trusted / review / flagged. The override log you’re building today is the training set.
later
Hosted sandbox at rubric.dev
Shareable URLs for prompt bakeoffs. Anonymous try-it, fork-to-local. Nothing ships until the local CLI has real users using it weekly.
