The problem

Iterating on a prompt is guessing.

We have git diff, test --watch, REPLs, linters. For prompts we have… a ChatGPT tab open in another window and a feeling.

Today

Edit, spot-check, ship

  • Tweak candidate.md. Paste it into a tab. Try three examples.
  • Merge. Hope.
  • Discover the regression via a Slack screenshot weeks later.
  • No idea whether yesterday's version was better or worse than today's.
With rubric

Edit, run, coach, promote

  • Edit. A real dataset runs against both baseline and candidate.
  • See which cases flipped and why.
  • A coach reads the losses and suggests specific edits to try.
  • When candidate wins, promote it. Git records the trajectory.
The workbench loop

One cycle. Measurably better every day.

You sit down at your workspace, run the cycle two or three times, promote once, close the lid. The prompt just got better. Do it daily for a month and the curve is compounding.

01 Edit Tweak candidate.md in the Prompts pane.
02 Run Sweep candidate vs. baseline across your dataset.
03 Coach Judge model reads the losses. Proposes specific edits.
04 Apply One click inserts the suggestion. Re-run.
05 Promote Candidate beats baseline. Swap. Git commits the trajectory.
Setup · under two minutes

Three commands. No account.

Single-file binary. No Node, no Python, no login wall. Edit prompt files on disk; rubric is a thin layer over the eval engine.

01
Download the binary.
# macOS / Linux — resolves arch automatically
curl -L github.com/rubric/rubric/releases/latest/download/rubric-$(uname -s).tar.gz | tar -xz
02
Scaffold a workspace.
./rubric init
wrote rubric.config.json
wrote prompts/baseline.md, prompts/candidate.md
wrote data/cases.jsonl
03
Open the workbench.
./rubric serve # three-pane UI on :5174

The feature that changes everything

The judge stops being a referee.
It becomes a coach.

After a sweep, the coach reads all the losses together, spots the pattern, and proposes concrete edits to candidate.md.

Not "the candidate needs improvement." Specific: "Case 1: you suggested 'nullable with a default' up front, which can rewrite/lock tables in some engines. Add a migration-safety rule."

One click appends the suggested edit into your candidate prompt. Save, re-run. The loss either flips, or it doesn't — either way you learn.

01 Avoid risky migration guidance
Case 1: candidate suggested adding a DEFAULT on ALTER TABLE. Baseline explicitly avoided it and won on safety.
For large-table schema changes: never recommend adding
a column with a DEFAULT in the same ADD COLUMN step.
Add NULLable → backfill → enforce NOT NULL separately.
Apply to candidate
02 Make the example optional when it crowds out the answer
Cases 4 and 5: the question was answerable in one line. The forced 3-part template made the response lose on brevity.
If the question can be fully answered by a single
factual statement, omit section 3 and keep the response
to 1–2 short sentences.
Apply to candidate

The judge isn't the final word

When you disagree, say so.

The judge is just another LLM. When you think it got a cell wrong, one click records the override — and that log becomes the training corpus for v2.3 calibration.

Works from the CLI and inline in the workbench. Same log either way: ~/.rubric/overrides/<project>.jsonl

rubric disagree case-3/openai/gpt-5.2 \
  --verdict A --reason "judge missed the factual error in B"
✎ case-3/openai/gpt-5.2 → A — "judge missed…"
Or inline, in serve
Your verdict Baseline Candidate Tie
✎ You → Candidate · "judge nailed it"
After you promote

CI protects the bar.

Once the prompt is good, tag the commit and let GitHub Actions gate future PRs against regressions. The PR comment shows top regressions inline.

The Action installs the release binary, takes ~15s to set up, and exits with code 2 when candidate loses more than it wins.
